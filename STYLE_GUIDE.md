# Style Guide

This document defines the writing and formatting conventions for
*Modern Operating Systems*. Consistency across chapters is critical
when many people contribute to a single book.

---

## General Principles

1. **Clarity over cleverness.** Write for a graduate student who is
   reading this chapter for the first time.
2. **Concrete over abstract.** When explaining a concept, show the
   command, the output, or the data structure — not just the idea.
3. **Measure, don't just describe.** If you claim fsync is slow, show
   the numbers. Every claim about system behavior should be backed by
   observable evidence.
4. **Respect the reader's time.** Say it once, say it precisely, move
   on.

## Markdown Conventions

### Headings

```markdown
# Chapter N: Title                    ← H1, one per file (chapter title)
## N.1 Section Title                  ← H2, numbered sections
### Subsection Title                  ← H3, unnumbered subsections
#### Rarely needed                    ← H4, avoid if possible
```

- Use exactly **one H1** per Markdown file.
- Number H2 sections as `N.1`, `N.2`, etc., where `N` is the chapter
  number.
- Do not skip heading levels (no H1 → H3).

### Text formatting

| Element | Syntax | Example |
|---------|--------|---------|
| OS/tool names | plain text | Linux, Ubuntu, Kubernetes |
| Commands and syscalls | backtick code | `perf stat`, `fsync()` |
| File paths | backtick code | `/proc/meminfo` |
| Key terms (first use) | **bold** | **vruntime** |
| Book/paper titles | *italic* | *Operating Systems: Three Easy Pieces* |
| Emphasis | *italic* | *not* the same thing |

### Code blocks

Always specify the language for syntax highlighting:

````markdown
```c
int main(void) {
    printf("hello\n");
    return 0;
}
```
````

For shell commands, use `bash` and include the prompt symbol:

````markdown
```bash
$ perf stat -e cache-misses ./qs random 1000000
```
````

For command output, use `text`:

````markdown
```text
 Performance counter stats for './qs random 1000000':
     12,345,678      cache-misses
```
````

### Callout boxes

Use blockquote-style callouts for special content:

```markdown
> **Note:** Supplementary information that is helpful but not
> essential to understanding the main text.

> **Warning:** Something the reader must be careful about — a
> common mistake, a safety concern, or a destructive operation.

> **Key insight:** A critical conceptual point that ties the
> section together. Use sparingly.

> **Deep dive:** Optional advanced material that can be skipped
> on first reading without losing the chapter's main thread.
```

### Lists

- Use `-` (hyphen) for unordered lists, not `*` or `+`.
- Use `1.` for ordered lists (let Markdown auto-number).
- Indent nested lists with 4 spaces.

### Tables

Use Markdown tables for structured data. Align columns for readability
in source:

```markdown
| Metric   | Baseline | Contended | Mitigated |
|----------|----------|-----------|-----------|
| p50 (us) |       12 |        15 |        13 |
| p99 (us) |       45 |     1,200 |        62 |
```

### Cross-references

Reference other chapters with relative links:

```markdown
As we saw in [Chapter 4](../../part2-process-scheduling/ch04-processes-threads/index.md),
context switches are triggered by...
```

Reference figures within the same chapter:

```markdown
![Memory hierarchy pyramid](figures/memory-hierarchy.png)

As shown in the figure above, the L1 cache...
```

## Figures

### File format

- **Diagrams:** SVG preferred (scalable, diffable in git). Export from
  draw.io, Excalidraw, or similar tools.
- **Screenshots:** PNG at 2x resolution (e.g., 1600px wide for a
  figure that renders at 800px).
- **Plots/graphs:** SVG or PNG generated by matplotlib/gnuplot with
  readable axis labels.

### Naming

Use lowercase, hyphen-separated, descriptive names:

```
figures/cfs-vruntime-tree.svg
figures/page-cache-writeback-flow.png
figures/lab3-baseline-vs-contended.png
```

Do not use generic names like `fig1.png`, `diagram.png`, or
`screenshot.png`.

### Alt text

Always include alt text that describes the figure's content:

```markdown
![Red-black tree showing four tasks ordered by vruntime, with the leftmost node highlighted as next-to-run](figures/cfs-vruntime-tree.svg)
```

### Captions

Add a caption line immediately after the image:

```markdown
![Alt text](figures/name.svg)
*Figure N.X: Caption describing the figure's significance.*
```

## Lab Sections

### Tiered difficulty

Labs use a three-tier structure:

- **Part A (Basic):** Required. Covers the minimum learning objective.
  Should be completable by all students.
- **Part B (Intermediate):** Required. Adds depth — multiple
  measurements, comparisons, evidence chains.
- **Part C (Advanced):** Optional. Extended exploration for motivated
  students. Often involves additional tools or custom experiments.

### Reproducibility requirements

Every lab must specify:

1. **Environment:** Ubuntu version, kernel version, required packages
2. **Setup:** Exact commands to prepare the environment
3. **Execution:** Step-by-step commands that produce results
4. **Expected output:** What the student should see (approximate ranges
   are fine — exact numbers vary by hardware)

### Data presentation

When a lab produces measurements, require students to present:

- A table with p50, p90/p95, and p99 percentiles
- The number of samples (N) and how variance was handled
- The specific `perf`/tool command used to collect the data

## Terminology

Use these terms consistently throughout the book:

| Preferred | Avoid |
|-----------|-------|
| container | Docker container (unless Docker-specific) |
| cgroup v2 | cgroups, cgroup v1 (unless comparing versions) |
| Kubernetes, K8s | k8s (lowercase), Kube |
| tail latency | long-tail latency |
| p99 latency | 99th percentile latency (after first use) |
| scheduling latency | scheduler delay, runqueue wait |
| vruntime | virtual runtime (after first definition) |
| eBPF | BPF, bpf (lowercase) |
| etcd | Etcd, ETCD |
| fsync | fSync, Fsync |

Define technical terms in **bold** on first use, then use them without
formatting thereafter.

## Citations and References

### Within the text

Use inline references with author-year style:

```markdown
The Completely Fair Scheduler replaced the O(1) scheduler in
Linux 2.6.23 (Molnar, 2007).
```

### Further Reading sections

At the end of each chapter, list references in a consistent format:

```markdown
## Further Reading

- Arpaci-Dusseau, R. H. & Arpaci-Dusseau, A. C. (2018).
  *Operating Systems: Three Easy Pieces.* Chapter 26: Concurrency.
  Available at https://pages.cs.wisc.edu/~remzi/OSTEP/
- Gregg, B. (2020). *Systems Performance*, 2nd ed. Addison-Wesley.
  Chapter 6: CPUs.
```

## Language and Tone

- Write in **second person** for labs and instructions: "Run `perf stat`
  and observe the cache-miss count."
- Write in **third person or first person plural** for exposition: "The
  scheduler selects the task with the smallest vruntime" or "We can
  observe this with `perf stat`."
- Avoid hedging ("it seems like", "it might be"). Be direct.
- Avoid jargon without definition. If you use a term for the first time,
  define it.
- Use American English spelling.
