# Lab: PagerUSE Oncall Simulation

> **Estimated time:** 3–4 hours
>
> **Prerequisites:** Chapter 3 concepts (percentiles, the USE method,
> evidence chains); Node.js 18+ installed in your VM
>
> **Tools used:** `top`, `free`, `vmstat`, `iostat`, `dmesg`, `ss`,
> `ps`, `cat /proc/vmstat`, `/usr/bin/time -v`

## Objectives

- Apply the USE method to diagnose tail latency problems in a
  simulated oncall scenario
- Practice forming hypotheses, collecting evidence, and ruling out
  alternative causes
- Produce a diagnosis report with an evidence chain of at least two
  independent signals plus one exclusion check

## Background

PagerUSE is a scenario-based debugging exercise: you are dropped into
a simulated incident with a pager alert, a context panel, and a
restricted observation "terminal". The terminal does not run commands
on your real machine — outputs are generated by the scenario, either
from fixtures or from a constrained LLM. Your job is not to guess;
it is to apply the USE method to narrow the search, then build an
evidence chain that supports your root-cause claim.

Three scenarios, in order of difficulty:

| Difficulty | Scenario | Main mechanism |
|---|---|---|
| Easy | `easy_memory_major_faults` | Memory pressure → major faults → p99 |
| Medium | `medium_multi_cause` | Paging **and** CPU runqueue bursts (two co-occurring mechanisms) |
| Hard | `hard_misleading_network` | Looks like network timeouts; actually storage tail + fsync + retry amplification |

Source for the simulator lives in `code/ch03-pageruse/`.

## Setup

Inside your Ubuntu VM:

```bash
sudo apt update
sudo apt install -y nodejs npm

cd code/ch03-pageruse
npm install

# Offline, deterministic mode (recommended for grading):
LLM_MODE=mock npm start
# Or, optionally, with an LLM backend:
# export OPENAI_API_KEY=... ; npm start
```

Open <http://localhost:3000>. You will see:

- **Active Alerts** (left) — the pager panel
- **Terminal** (right) — the constrained observation surface
- **Context** and **Student Task** (bottom)

### Terminal rules (important)

- The terminal does **not** execute commands on your machine.
- Only read-only observation commands are allowed: `top`, `free`,
  `vmstat`, `iostat`, `cat`, `grep`, `dmesg`, `journalctl`, `ss`,
  `ps`, and a few more. Each scenario lists its allowlist under
  `help`.
- Shell features are disabled: no pipes (`|`), redirects (`>`),
  chaining (`;`, `&&`), command substitution (`$(...)`), or quoted
  strings. This is intentional — the point is to practice reasoning
  about OS mechanisms, not shell cleverness.

## Part A: Easy Scenario — Single Root Cause (Required)

**Scenario:** `easy_memory_major_faults`

**Goal:** Identify the memory-pressure mechanism and build a clean
evidence chain.

### A.1 Read the Alert

Write down in your report:

- The symptom (what is the p99? what is the baseline?)
- The blast radius (what fraction of requests?)
- The time window (when did it start?)
- Any context hints (recent deploys, config changes, traffic shifts)

### A.2 Frame with USE

Before investigating, list one hypothesis per resource:

| Resource | Utilization question | Saturation question | Errors question |
|---|---|---|---|
| CPU | Is CPU% elevated? | Is there a runqueue? | Throttling? |
| Memory | Is RSS near the limit? | Reclaim activity? | Major faults or OOM? |
| Disk | Is the disk busy? | Queue depth? | I/O errors? |
| Network | Bandwidth? | Connection backlog? | Retransmits? |

Pick the two most plausible hypotheses. You will collect evidence for
one and an exclusion signal for the other.

### A.3 Collect Evidence

Issue observation commands through the terminal. Aim for:

- At least **two independent signals** that support your primary
  hypothesis. "Independent" means they measure different things —
  `pgmajfault` from `/proc/vmstat` and `Major page faults` from
  `/usr/bin/time -v` are independent; two runs of `top` are not.
- At least **one exclusion check** that rules out a plausible
  alternative.

For the memory-pressure case, a good set looks like:

1. `free -m` — available memory is near zero.
2. `cat /proc/vmstat` — `pgmajfault` is far above baseline.
3. `top -bn1` — CPU% is moderate; wait time (`wa`) is non-trivial.
4. `iostat -x 1 5` (exclusion) — disk queue depth and `%util`
   flat except for brief bursts. Rules out "disk is the primary
   bottleneck".

Copy the key lines — not the full dump — into your `transcript.txt`.
For each line, write one sentence on why it matters.

### A.4 Write the Diagnosis

In `report.md`, follow the template:

- **Symptom.** p99 rose from X to Y at time Z, affecting ~N % of
  requests.
- **Hypothesis.** Memory pressure → major faults → tail spike.
- **Signal 1.** (cite the specific line; say what it shows)
- **Signal 2.** (independent; say what it shows)
- **Exclusion.** (what you ruled out, and what signal ruled it out)
- **Mechanism.** 1–2 paragraphs. Why does a major fault inflate p99
  but not p50? What queue does the request wait in?
- **Mitigation.** What would you change? (Raising the memory limit,
  reducing working set, pinning hot data, etc.)
- **Verification plan.** At least two checks: p99 recovery and a
  mechanism-aligned signal (e.g., `pgmajfault` returns to baseline).

### A.5 Part A Checklist

- [ ] USE checklist filled in
- [ ] Two independent supporting signals
- [ ] One exclusion check
- [ ] Mechanism explained in terms of an OS slow path
- [ ] Mitigation + two verification checks

## Part B: Medium Scenario — Multiple Contributing Causes (Required)

**Scenario:** `medium_multi_cause`

**Goal:** Separate two co-occurring mechanisms (paging + CPU runqueue
bursts) and explain how each contributes to the tail.

The complication: each mechanism alone would produce a tail, and both
are present. You must:

1. Identify both mechanisms with evidence.
2. Argue which one dominates (or that they compose), citing
   measurements.
3. Propose a mitigation for each, and decide which to try first and
   why.

Good reports will include a small table mapping each observed signal
to the mechanism it supports, so the reader can see at a glance that
the diagnosis is coherent. Weak reports will either pick one cause
and ignore the other, or list both without connecting them.

Key observation targets:

- Memory side: `/proc/vmstat pgmajfault`, `free -m`, `dmesg` for
  reclaim / OOM hints.
- CPU side: `cat /proc/loadavg`, `vmstat 1 5` (`r` column for
  runqueue, `cs` for context switches), `top -bn1` (`%wa` and
  `%system`).
- Exclusion: `iostat -x 1 5` to rule out disk as primary.

### Part B Checklist

- [ ] Both mechanisms identified with separate signals
- [ ] Explicit discussion of how they compose
- [ ] Mitigation ordering argued (which to try first and why)

## Part C: Hard Scenario — Misleading Symptom (Optional)

**Scenario:** `hard_misleading_network`

**Goal:** Resist the obvious suspect (network) and follow the
evidence to the actual cause (storage tail + fsync + retry
amplification).

The symptoms look like network timeouts. Naïvely, the story is "the
network is slow". Your task is to:

1. Find at least one signal that *disconfirms* the network
   hypothesis (e.g., `ss -s` shows stable connection counts and no
   retransmits; the "timeout" is from an application-layer retry
   loop, not TCP).
2. Identify the actual mechanism (storage-side fsync latency
   amplified by retries).
3. Explain how the retry loop converts a small set of slow commits
   into a large apparent timeout rate.

This scenario rewards reading the application log alongside system
counters — the root-cause chain is longer, and the final mechanism
(retry amplification) is *composed* of OS-level and
application-level behavior.

### Part C Checklist

- [ ] Disconfirming signal for the misleading hypothesis
- [ ] Correct mechanism identified with supporting evidence
- [ ] Explanation of retry amplification

## Deliverables

Submit a directory containing:

1. **`report.md`** — the core write-up, one section per scenario you
   attempted. For each scenario: symptom, hypothesis, two supporting
   signals, one exclusion, mechanism, mitigation, verification plan.
2. **`use_checklist.md`** — filled in (copy from
   `code/ch03-pageruse/use_checklist.md` as a starting point).
3. **`transcript.txt`** — the terminal commands you issued and the
   output lines you cited, with brief annotations.

Your `transcript.txt` should look like:

```text
# scenario: easy_memory_major_faults
student@host:~$ free -m
              total        used        free      available
Mem:          1024         960          10             45
# -> memory utilization near the limit; available is almost zero

student@host:~$ cat /proc/vmstat
pgmajfault 34127
# -> 10x baseline; primary signal for memory-slow-path
```

## Common Pitfalls

- **"CPU is not high, so it is not a CPU problem."** Utilization and
  saturation are different. Runqueue bursts can happen with moderate
  `top` readings.
- **"Disk `%util` is not 100 %, so it is not storage."** Tail latency
  comes from *some* slow I/O, not average saturation.
- **One signal = diagnosis.** It is a hypothesis. You need two
  independent signals plus an exclusion.
- **Full dumps instead of key lines.** Pasting the entire output of
  `top` is not evidence. Pointing at one line and saying why it
  matters is evidence.
